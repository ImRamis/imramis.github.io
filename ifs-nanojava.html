Key Concepts
  • Security Level: A label (e.g., LOW or HIGH) indicating the confidentiality of data.
  • Lattice: A model defining allowed information flows. The simplest is LOW ‚â§ HIGH, meaning information can flow from LOW to HIGH, but not vice-versa.
  • Explicit Flow Violation: A direct assignment of HIGH-security data to a LOW-security variable. Ex: lo = hi;
  • Implicit Flow Violation: An indirect leak where HIGH-security data influences the program's control flow (e.g., an `if` statement) to affect a LOW-security variable. Ex: if (hi > 5) { lo = 1; }
  • Program Counter (pc): A security level representing the context of the code currently executing. It becomes HIGH inside `if` or `while` blocks that depend on HIGH data.
  • Taint Tracking: The process of monitoring how security levels propagate through variables as the program executes.
Formal Typing Rules (for Q2)

The analyzer uses these rules to determine if a program is secure.

  • (TAss): For an assignment `x = e;` to be secure, the security level of expression `e` (plus the current `pc` context) must be less than or equal to (`‚â§`) the level of variable `x`.
  • (TIf): For an `if(b){P1}else{P2}` statement, the level of the condition `b` raises the `pc` for `P1` and `P2`. The security of the whole `if` statement is the lowest level of `P1` and `P2`.
  • (TWhile): For a `while(b){P}` loop, the level of the condition `b` must be less than or equal to (`‚â§`) the security level of the body `P`. The level of `b` also raises the `pc` for the body.
  • (TSeq): For a sequence `P1; P2`, the security of the whole sequence is the lowest level of `P1` and `P2`.
  • (TSkip): The `skip;` command is always secure. It does nothing, so it cannot cause a leak.
